# 《一次买够》后端下一阶段任务说明

## 一、阶段定位
本阶段目标是从“演示可用”推进到“业务可用与工程可用”，在不引入过重基础设施的前提下，完成数据持久化、安全强化、稳定性治理与核心流程可验证建设。

## 二、阶段目标（总览）
1. **数据与状态可持续**：核心业务数据落地数据库，重启不丢失。
2. **鉴权与安全可控**：令牌机制可过期、可刷新、可撤销。
3. **接口稳定可联调**：参数校验清晰、错误码统一、日志可追踪。
4. **关键流程可验证**：核心链路具备最小可回归测试集。

---

## 三、任务拆解（按优先级）

### P0-1 数据持久化改造（必须优先）
**目标**：将内存数据替换为数据库存储，保证用户、收藏、提醒、历史等核心数据长期可用。

- **关键实现**
  1. 设计核心表（用户、商品、收藏、提醒、浏览历史、搜索记录）。
  2. 建立索引与主外键，确保搜索、收藏、提醒查询性能。
  3. 数据层改造：将 `app/data.py` 的内存逻辑迁移为数据库访问层（DAO/Repository）。
- **产出物**
  - 数据库表结构说明（建表 SQL 或迁移说明）
  - 数据访问层实现与接口替换记录
- **验收口径**
  - 服务重启后数据仍可读取
  - 主要接口（收藏/提醒/历史）正确读写数据库

### P0-2 认证与安全升级（必须优先）
**目标**：从演示态 Token 升级为可过期、可刷新、可失效的生产级鉴权机制。

- **关键实现**
  1. 明确令牌策略（JWT 或会话 + 黑名单）。
  2. 实现 access/refresh 过期机制与刷新流程。
  3. 登录/登出后令牌失效策略与黑名单治理。
- **产出物**
  - 鉴权流程说明（token 生命周期、刷新与失效策略）
  - 鉴权中间件与统一依赖
- **验收口径**
  - 过期 access_token 无法访问
  - refresh_token 能获取新 access_token
  - logout 后旧 token 不可用

### P0-3 接口稳定性与错误治理（必须优先）
**目标**：减少前端适配成本，保证错误处理一致且可定位。

- **关键实现**
  1. 统一参数校验规则与错误码规范。
  2. 统一异常映射与结构化日志字段（trace_id、user_id、path）。
- **产出物**
  - 错误码清单与校验规则说明
  - 统一异常处理与日志规范说明
- **验收口径**
  - 关键接口错误返回结构一致
  - 日志可追踪至具体请求与用户

---

### P1-1 真实商品数据接入（高优先）
**目标**：将演示商品替换为真实或近真实的数据来源，提升结果可信度。

- **关键实现**
  1. 明确数据来源范围与更新策略。
  2. 建立数据质量校验规则（价格区间、平台字段、标题字段）。
- **产出物**
  - 数据接入说明（字段映射、更新频率）
- **验收口径**
  - 搜索结果可覆盖多平台且字段完整

### P1-2 降价提醒异步化（高优先）
**目标**：将“提醒触发”从同步逻辑迁移为异步任务，保证可扩展与可重试。

- **关键实现**
  1. 定时任务扫描提醒规则并触发通知。
  2. 失败重试与触发状态记录。
- **产出物**
  - 提醒任务流程说明
- **验收口径**
  - 满足触发条件的提醒能被标记为已触发并记录时间

### P1-3 测试体系完善（高优先）
**目标**：为搜索、收藏、提醒三条核心链路建立可回归测试集。

- **关键实现**
  1. 增加关键接口的单元/集成测试。
  2. 建立最小回归清单。
- **产出物**
  - 测试用例与覆盖范围说明
- **验收口径**
  - 测试可稳定运行并覆盖核心链路

---

### P2-1 推荐能力增强（持续优化）
**目标**：逐步替换演示态推荐逻辑，接入基础行为特征。

- **关键实现**
  1. 明确行为事件口径与特征字段。
  2. 推荐策略可配置、可迭代。
- **产出物**
  - 推荐策略说明与字段定义

### P2-2 可观测性建设（持续优化）
**目标**：提升线上问题发现与定位能力。

- **关键实现**
  1. 结构化日志、指标与告警规则。
  2. 关键接口耗时监控。
- **产出物**
  - 日志字段规范与监控指标说明

### P2-3 工程化与部署标准化（持续优化）
**目标**：推动容器化部署与 CI/CD 质量门禁。

- **关键实现**
  1. 环境配置分层与部署脚手架。
  2. 引入构建、测试与静态检查的质量门禁。
- **产出物**
  - 部署与 CI/CD 说明

---

## 四、风险与依赖
1. 数据接入依赖外部平台稳定性与字段一致性。
2. 持久化改造可能影响接口响应与字段兼容性。
3. 鉴权升级需要前端配合更新 Token 逻辑。

---

## 五、阶段输出建议
- 保持“文档 + 可运行代码 + 最小测试”三件套同步输出。
- 所有变更均以 `openapi.json` 为契约基准，确保前后端一致。
