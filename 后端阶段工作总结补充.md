## 二、后端框架完善补充（在现有实现基础上的结构化说明）

### 2.1 分层与职责边界补充
当前后端演示版框架已具备清晰的职责分层，补充说明如下：

1. **接入层（运行与承载）**
   - Uvicorn 作为 ASGI 运行入口；生产态可替换为 Nginx + Gunicorn/Uvicorn 多进程模式。

2. **接口层（API 路由与业务编排）**
   - `app/main.py` 统一定义路由与业务入口，保持“认证、商品、用户、收藏、提醒、历史、推荐、埋点”的模块化分组。

3. **领域与数据层（模型与数据访问）**
   - `app/schemas.py` 统一接口模型与响应结构（`ApiResponse` + DTO）。
   - `app/data.py` 作为演示态数据访问层，集中封装查询、写入、分页、排序与统计逻辑。

4. **横切能力（统一治理）**
   - 统一响应与错误码结构，降低前端解析成本。
   - 统一鉴权依赖注入（`Depends(get_current_user)`）。
   - 统一分页结构，确保列表接口一致性。

### 2.2 框架可扩展路径补充
本阶段以演示闭环为主，但框架已预留扩展路线，补充说明如下：

- **数据持久化**：将 `data.py` 替换为持久层（MySQL/Redis），保持接口层无感。
- **任务与异步**：降价提醒可演进为定时任务/消息队列，推荐与埋点可落入异步消费。
- **认证升级**：从内存 Token 过渡至 JWT/会话机制，补充黑名单、刷新策略与安全审计。
- **可观测性**：补齐结构化日志、指标与告警，保留统一错误码口径。

## 三、前后端对接可能问题与建议

### 3.1 鉴权与令牌流转
- 登录接口返回 `access_token / refresh_token`，需前端持久化并按需刷新。
- 统一使用 `Authorization: Bearer <token>` 传递；401/403 应按统一错误码处理。
- 演示版 Token 为内存态，**重启即失效**，联调时需避免缓存过久。

### 3.2 统一响应与错误处理
- 所有接口均返回 `ApiResponse` 结构，前端解析应以 `code` 为首要成功/失败判断依据。
- 失败场景需同时关注 HTTP 状态码与业务错误码（详细字段以 `openapi.json` 为准）。

### 3.3 分页与排序参数一致性
- 列表接口使用 `page/page_size` 规范；前端需确保请求值与响应 `pagination` 结构一一对应。
- 排序字段（如 `price_asc/price_desc/sales_desc/rating_desc`）须与契约一致，避免传入未知值。

### 3.4 筛选条件与字段类型
- 搜索过滤（类目、平台、价格区间）需与 `openapi.json` 中字段类型一致。
- 对 `min_price/max_price` 的单位与精度建议前后端统一约定（演示版以数值为准）。

### 3.5 数据一致性与重复操作
- 收藏与提醒的“重复创建”在后端会被显式提示；前端需对重复场景做友好提示。
- 详情接口若 `product_id` 不存在，需做不存在提示或引导回退。

### 3.6 兼容与演示限制
- 现阶段为演示数据，部分字段可能为示例值；联调时请以 `openapi.json` 字段定义为最终依据。
- 与需求文档中“真实采集/实时比价”相关能力暂未实现，需在联调文案中明确“演示态”。
