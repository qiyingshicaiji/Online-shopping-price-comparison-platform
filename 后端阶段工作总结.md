# 《一次买够》网购比价平台后端阶段工作总结

## 一、文档说明

### 1.1 编写目的
本文件用于系统化总结当前阶段后端工作成果，明确已完成范围、关键问题与解决思路，并形成下一阶段可执行计划，便于项目汇报、团队对齐与后续实施跟踪。

## 三、遇到的问题与解决思路

### 3.1 接口风格不统一导致前端适配复杂

**问题表现：**  
不同接口若返回结构不一致，会导致前端需要分支解析，增加联调复杂度与缺陷风险。

**解决思路：**  
建立统一响应模型与统一异常处理机制，将成功与失败场景规范化，保证解析逻辑可复用。

**结果：**  
前端可通过统一判定规则消费接口，联调效率提高。

### 3.2 多条件搜索逻辑易耦合、易出错

**问题表现：**  
关键词、类目、价格、平台、排序、分页并存时，逻辑顺序不清晰容易导致结果异常或维护困难。

**解决思路：**  
明确“过滤 → 排序 → 分页”的固定处理链路，将搜索逻辑分层执行。

**结果：**  
行为更可预期，后续新增筛选条件时改动边界清晰。

### 3.3 鉴权逻辑与业务逻辑混杂

**问题表现：**  
若每个接口自行处理鉴权，会产生重复代码，并增加遗漏风险。

**解决思路：**  
通过依赖注入统一做用户身份校验，业务函数仅处理业务语义。

**结果：**  
代码职责更清晰，可读性与一致性提升。

### 3.4 演示阶段缺少真实数据源与持久化

**问题表现：**  
项目早期若直接引入完整采集链路和数据库，实施成本高、迭代慢，不利于先验证业务闭环。

**解决思路：**  
先采用内存示例数据完成接口契约落地和主流程贯通，待联调稳定后再替换为持久化方案。

**结果：**  
显著缩短了首轮可用版本交付时间，验证路径明确。

### 3.5 测试与运行环境初始依赖不完整

**问题表现：**  
在未安装测试依赖时，直接运行测试会出现 `No module named pytest`。

**解决思路：**  
统一依赖安装入口：`pip install -r requirements.txt`，再执行 `python -m pytest -q`。

**结果：**  
当前基础测试可通过（2 项通过），形成可复现的最小验证流程。

## 五、下一阶段计划（按优先级）

### 5.1 P0（必须优先）
1. **数据持久化改造**  
   将内存数据替换为数据库存储（建议 MySQL），建立核心表与索引，完成数据生命周期管理。

2. **认证与安全升级**  
   从演示态令牌升级为生产级 JWT/会话方案，补充过期策略、刷新策略、黑名单机制与鉴权中间件。

3. **接口稳定性与错误治理**  
   统一参数校验标准、错误码规范与异常日志结构，降低线上排障成本。

### 5.2 P1（高优先）
1. **真实商品数据接入**  
   完成采集/聚合链路接入，建立平台数据更新策略与数据质量校验机制。

2. **降价提醒异步化**  
   引入定时任务/消息机制，实现提醒触发扫描、通知下发与失败重试。

3. **测试体系完善**  
   扩展单元测试、集成测试与关键接口回归测试，提升改动安全性。

### 5.3 P2（持续优化）
1. **推荐能力增强**  
   基于用户行为与商品特征迭代推荐策略，逐步替换演示态逻辑。

2. **可观测性建设**  
   补齐结构化日志、核心指标监控、告警策略与链路追踪能力。

3. **工程化与部署标准化**  
   推进容器化发布、环境配置分层、CI/CD 质量门禁（测试、静态检查、构建）。

## 六、风险与依赖说明

### 6.1 主要风险
- 真实采集数据质量与时效不稳定，可能影响比价结果可信度。
- 持久化改造过程中数据模型调整，可能引发接口兼容风险。
- 提醒与推荐能力依赖行为数据质量，早期效果波动较大。

### 6.2 外部依赖
- 业务侧对“平台接入范围、频率、准确率”的明确约束。
- 前端对字段稳定性、错误码语义的一致遵循。
- 运维侧对部署环境、日志平台、监控平台的支持能力。

当前后端阶段已完成“可运行、可联调、可验证”的目标，核心业务链路和接口契约基本打通，具备进入下一阶段（持久化、真实数据接入与生产化治理）的实施条件。  
后续应优先推进 P0 事项，确保系统从“演示可用”平稳过渡到“业务可用、工程可用、运维可用”。
